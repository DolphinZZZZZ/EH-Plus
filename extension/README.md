# Extension Build

This directory contains the Manifest V3 1.0.0 extension build of EH＋. The default visible language is Chinese, and the language setting is applied to the floating panel and popup together.

Current behavior:

- Uses `service-worker.js` and `chrome.storage.local` for runtime state.
- Provides only a lightweight popup pointer. Account status, storage, statistics, Dawn/check-in, and settings are kept in the page floating panel.
- Settings are expanded inside the Page Preload-style floating panel instead of opening a separate options page.
- The extension injects a Page Preload-style draggable floating panel only on EH/EX `/g/` gallery pages and `/s/` reader pages and marks those pages as extension-owned.
- The floating panel follows the original Page Preload code shape (panel/head/actions/settings/matrix structure; the original `ehp-` class prefix was renamed to `ehplus-` in 1.0.0, e.g. `#ehplus-panel`).
- The service worker starts real reader preloading from EH/EX `/s/` pages and extracts reader links from `/g/` gallery pages. On `/s/` pages it reads real next and previous reader URLs, then preloads in alternating order: next page, previous page, second next page, second previous page, with `globalConcurrency` controlling how many reader page tasks can run at once. It stores preloaded reader image records in the active cache store: default IndexedDB `ehplus-preload-cache`, or the authorized custom directory when enabled. Authorizing a directory acknowledges the picker message immediately and copies existing IndexedDB cache records into that directory in the background. The directory store writes identifiable reader image files as `images/<gid>/<gid>-<pageNo>.<ext>`, keeps unclassified image files directly under `images/`, writes record indexes under `records/`, and mirrors state, settings, daily logs, and statistics under `state/`, `settings/`, `logs/YYYY-MM-DD.json`, and `stats/`. Opening the directory store also migrates existing flat `images/` files and old `images/<gid>/page-<gid>-<pageNo>.<ext>` names into the gallery-folder layout and updates the affected main/resource record `directoryImageFile` values. Directory summary refresh reads record JSON only and must not hydrate image files into data URLs; if listing times out, the previous storage counters are kept instead of being overwritten with zero. New reader preload work is skipped only when the preload switch is off or EH＋ built-in auto-pager is still continuing. Current-page image activity is checked immediately before each preload fetch: if another page image request is still active, that not-yet-sent fetch waits, while already-sent preload requests and other preload workers continue. Multiple open EH/EX pages preload concurrently and share one global request-slot pool sized by `globalConcurrency`; the focused tab's requests get free slots first, switching tabs reorders only not-yet-sent requests, and unknown focus state falls back to equal treatment so scheduling can never stall preloads (waiter timeouts force release as a failsafe). The engine also reconciles its unsent queue with page observations at most once per second: pages already loaded in the page become link-only follow-ups (no duplicate image fetch, chain discovery preserved), pages currently loading are downgraded to the queue tail, and a stale page session drains the queue; missing observations keep the default alternating order. Records stored as directory image files count as cached, so preload does not re-fetch images that already exist on disk.
- The built-in auto-pager enable switch is in the top quick toggles, while the floating panel's auto-pager settings group edits the tuning values. It follows Super-preloader's EH/EX `/s/` rule shape by reading `#img` and `#next` / `nexturl`, appending reader fragments with `sp-exhentai-img-*` IDs, and exposing `remain`, `maxpage`, immediate pages, separator, and Aplus-style local next-page preparation settings. It does not call the cooperative/page cache API; inserted pages use an internal service-worker message to read the active local cache store first, then fall back to the original EH/EX image URL on miss. With Aplus enabled (default), the controller follows Super-preloader's stitching-speed model: it prefetches the next reader page HTML and prebuilds the insert fragment (including the internal cache query) at start and after each append, so reaching the scroll threshold inserts the already-prepared fragment immediately; a failed prefetch falls back to fetching at the threshold. With Aplus disabled, the next page is only requested when the scroll threshold is reached. On start the controller first settles any pending reader cache-first query and waits for the tab-scoped DNR gate release before prefetching or stitching, so early inserted images are never redirected to the cache-first placeholder. Because images inside a detached prebuilt fragment fire their error events before insertion (where no window-level listener can see them), each append re-dispatches an `error` event on inserted reader images that already failed, handing them to the MAIN-world nl retry guard for in-place source replacement.
- The extension can consume that same local cache without issuing H@H image fetches on cache hits only when `优先取本地缓存` / `Prefer local cache` is explicitly enabled; the switch defaults off to preserve normal page access. The page-side holder only honors the current V2 enable hint and clears the legacy enable hint so an older `EHPLUS_READER_CACHE_FIRST_ENABLED=1` localStorage value cannot keep interception active after upgrade. When the V2 hint has never been written for an origin (fresh browser profile or cleared site data), the content script asynchronously reads the real setting from `chrome.storage.local` at startup; if enabled it rewrites the page hint for future navigations and performs a one-time late cache apply on the current page (replacing the image source on hit, doing nothing on miss) without re-installing the DNR gate or the placeholder-hold flow. `/s/` pages use a tab-scoped `declarativeNetRequest` gate, a `document_start` MAIN-world holder script, and the isolated content script: the gate redirects initial H@H image subresource loads to an extension placeholder while the isolated script queries the exact `gid:pageNo` cache. The placeholder visibly shows `正在读取本地缓存...` and animates the ellipsis from one dot to three dots in a loop. Hits keep the gate active for that reader page while the opt-in switch remains enabled and apply a deliverable local URL; misses, errors, the 3500 ms fast-query timeout, or setting-disabled release the gate before restoring or allowing the original URL, with a short release-timeout fallback so the page is not left on the placeholder. The longer hard timers remain only as stalled-message watchdogs. The controller exposes query diagnostics through page dataset fields for `send-start`, `send-resolve`, `send-error`, `response-hit`, and `response-miss`, including elapsed time, response reason/delivery kind, runtime errors, and `data-ehplus-cache-first-network-block-*` gate state. `/g/` pages still replace existing thumbnail images inside existing `/s/` links only when the exact reader-page cache entry is already present. When the switch is off, these replacement paths are skipped and the extension only observes loaded images for cache-fill. Per the 2026-07-06 planning update, the MAIN-world holder no longer probes third-party auto-pager DOM traces (Super-preloader, Pagetual, AutoPagerize, Infy Scroll); it only skips hold/apply when EH＋'s own built-in auto-pager `data-ehplus-autopager` markers are present. The isolated controller uses the same own-auto-pager DOM detection (`hasBuiltInAutoPagerDom`), so when the built-in auto-pager is active both worlds yield together: the controller settles as `auto-pager`, releases the DNR gate, and skips reader/gallery cache application.
- The floating panel has a compact collapsed state that keeps only the status dot, project title/update badge, and Expand button visible. Double-clicking the left-side header area, including the status dot and project title but excluding buttons, also toggles the panel.
- The floating panel includes a colors group for previewing every status/matrix dot color via dot click, color picker, or hex input; Confirm applies and saves, while Cancel or clicking elsewhere discards the pending edit.
- The floating panel includes an about group with current version, GitHub icon/link, GitHub-only source notice, free/open-source notice, and paid-copy resale warning.
- The GitHub project link opens the real project repository. Update checks query GitHub Releases, default to a 7-day automatic interval, and show a clickable title notice when a newer release is available.
- Account refresh reads real `https://e-hentai.org/home.php`, `https://e-hentai.org/exchange.php?t=hath`, and `https://e-hentai.org/exchange.php?t=gp` pages with browser cookies. ExHentai pages can trigger the refresh, but account endpoints stay on E-Hentai.
- Manual Dawn checks read real `https://e-hentai.org/news.php`; `It is the dawn of a new day!` is parsed as the daily successful check-in and keeps reward details in the result display, while a successful non-Dawn response such as the HentaiVerse monster event is treated as already checked today and displays the check date.
- Image-limit reset uses a confirmation step, then posts the real `reset_imagelimit=Reset+Quota` body to `https://e-hentai.org/home.php` and rereads balances to calculate the actual delta.
- Statistics are always recorded in the backend. The statistics tab only controls frontend display; by default, no statistics fields are selected for the top mini summary.
- On `/s/` reader pages the floating panel shows a live image-loading line under the main status line with the latest two events (loading page n, page n served from local cache, page n loaded, page n failed and retrying with a new source, retry succeeded/still failing), covering both the current reader image and built-in auto-pager inserted images; it hides itself after 8 seconds without events and never displaces the cache-first hit message on the main status line.
- Directory migration (authorize-time, manual start/retry, and directory-to-directory switch) runs in the background with per-entry verification: every copied record is re-read from the target, its index must resolve and its image file must hydrate with matching byte size; directory-to-directory migration hydrates real image files from the source before writing so the new directory receives image bodies, not just index JSON. Entries already present and equivalent in the target are skipped, which makes a retry resume from where a failed, cancelled, or interrupted run stopped. The migration tab shows live `migrated x / y (skipped z)` progress with 1-second polling and offers a Cancel button (`EHPLUS_CANCEL_MIGRATION`); cancelled or failed runs never delete the old cache.
- The cleanup tab greys out both the Estimate and Confirm buttons while the days input is invalid; both recover together once the input is valid again.
- Browsing history records gallery/reader visits as lightweight `recordKind: 'history'` entries in the active cache store, deduplicated by gid and capped by the configurable history limit (default 100). A standalone `history.html` page (opened from the floating panel statistics tab or the toolbar popup, always through the service worker `EHPLUS_OPEN_HISTORY` message because the page is not web-accessible and page-context `window.open` would be blocked by Chrome) lists entries EhViewer-style with thumbnail, category, uploader, rating, pages, visit count, and last visited time, supports source filtering (`/g/`, `/s/`, or all), sorting by last visited or visit count, per-entry delete, and clear-all. History entries never join cache cleanup, image limits, dedupe, cooperative cache hits, or `/s/` hit-rate statistics. Card covers use the gallery default cover captured on `/g/` visits; `/s/` visits reuse the `/g/` cover already merged onto the same gid entry, and when no gallery cover exists the history page falls back to the locally cached first reader page (`gid:1`) through the internal cache query without issuing any page or image network request, keeping the `EH` placeholder on miss.
- Exposes the optional `EHPLUS_COOPERATIVE_CACHE_QUERY` message for cooperative cache reads. Other extensions can use `chrome.runtime.sendMessage`; page scripts on E-Hentai / ExHentai can use `window.postMessage` and receive `EHPLUS_COOPERATIVE_CACHE_RESPONSE` with the same `requestId`. Queries can identify exact reader, gallery, or resource cache entries by `pageKey`, `/s/` page URL, `galleryKey`, `/g/` gallery URL, or image/resource URL. The current build can return preloaded reader images as data-URL delivery records; misses only return miss metadata and do not trigger page/image requests.
- Logs default to the most recent 30 days and 100 MB, and entries include event, action, request ID, source, sender page context, and structured details.
- Marks injected EH/EX pages as extension-owned.
- Runtime state is stored under `ehplus_live_state`; old `ehplus_offline_state` data is removed on install/update so fake account/cache/log data from earlier test builds is cleared. Fixture files under `fixtures/` remain because they are used by automated tests.

Build and copy to Downloads:

```powershell
npm run build:extension
```

The build excludes design assets from the shipped package (`icons/candidates/`, `icons/icon-1024.png`, `icons/icon-applied-preview.png`, ~21 MB combined); only the manifest-referenced 16/32/48/128 icons ship. The design assets stay in the source repository and are backed up separately as an icon-candidates backup zip in Downloads.
