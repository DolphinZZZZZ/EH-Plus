# Phase 1 Local Test Plan

Status: completed. The current extension is version 1.0.1 for EH/EX. This document remains as the historical boundary for the first local simulation milestone; current behavior is described in `README.md` and `extension/README.md`.

Phase 1 intentionally avoids real browser and network effects.

## Hard Boundaries

- Do not open or automate E-Hentai / ExHentai pages.
- Do not trigger real Dawn of a New Day events.
- Do not submit image-limit reset requests.
- Do not build or exercise a Tampermonkey/userscript package.
- Do not write to Chrome, Edge, or Firefox profiles.
- The only allowed real network path in the Phase 1 local test extension was the future GitHub Release update check/download flow. The current 1.0.1 build uses the configured project Release API instead of forcing a simulated update.

## Local Fixtures

- `fixtures/images/`: generated PNG test images containing `测试图片N`.
- `fixtures/html/`: account and Dawn parsing samples, including first-check and already-checked responses.
- `fixtures/data/`: gallery and cache records.

## Verification

Use `npm test` to run the offline test suite.

Use `npm run build:extension` to create the Chrome MV3 package under `dist/` and copy it to the user's Downloads folder:

`EH＋-extension-v<manifest version>`

Load that folder with Chrome developer mode. The popup only points users back to the page floating panel. Account status, cache, statistics, Dawn/check-in, and settings live in the page floating panel. The current extension performs real E-Hentai account/Dawn requests and EH/EX reader-page preloading when loaded on EH/EX pages; automated tests still use local fixtures.

Earlier Phase 1 local UI testing allowed injection on ordinary `http` and `https` pages. The current 1.0.1 extension no longer does that: account, Dawn, and quota requests use E-Hentai account pages, ExHentai permission supports page injection and EH/EX gallery/reader preloading, and `content_scripts.matches` are limited to EH/EX `/g/` gallery pages and `/s/` reader pages.

The injected page UI should reference the original Page Preload source structure and behavior: draggable `#ehp-panel`, `.ehp-head`, `.ehp-actions`, `.ehp-settings`, `.ehp-matrix`, settings action, quick toggles, status lines, and matrix dots. On reader pages, matrix dots should use the current `gid:pageNo` and bounded reader cache summaries for exact cached/index-only status; only fall back to aggregate storage and queue counts when exact records are unavailable. The test build keeps the UI behavior but uses local simulation instead of real preloading/network/cache writes. It must not expose top-level test buttons that trigger simulated requests and write log entries.

Settings must expand inside that floating panel, like the original script UI model. Do not open a separate Chrome options page for the normal settings workflow. The Settings button should sit at the lower-right of the expanded panel, mirroring the upper-right collapse button spacing.

The floating panel has a compact collapsed state. When collapsed, it keeps only the status dot, project title/update badge, and Expand button visible so the test UI does not keep covering the page. Double-clicking the left-side header area, including the status dot and project title but excluding buttons, must also expand or collapse the panel. That header double-click path has a 1-second cooldown after it toggles; the right-side Expand/Collapse button must not be throttled by that cooldown.

When the viewport is too short for the active tab content, the floating panel caps its height to the visible page and supports vertical scrolling, so lower settings remain reachable.

Settings should apply immediately after each change. A manual sync/save button may remain only as a fallback to re-send the current settings.

The default visible language is Chinese. The language setting must apply to the floating panel, settings drawer, status lines, copied diagnostics, and popup message together. English mode should not leave the top action buttons or quick settings in Chinese, and Chinese mode should not leave them in English; internal code identifiers may remain English for compatibility with the original Page Preload structure.

Log retention and size limit must be real settings, initialized to 30 days and 100 MB in their controls rather than described only in helper text. Log entries should be detailed enough for troubleshooting, including event name, action, request ID, simulated flag, sender page context, and structured details.

The current floating panel includes groups for storage statistics, image-cache controls, log settings, account status and quota reset confirmation, Dawn, `/s/` and `/g/` statistics, cleanup preview/confirm, authorized-directory migration, language, colors, and about/update information. Its statistics group keeps field visibility and history-limit controls only, and opens the standalone browsing history page without embedding a frequently watched list. It does not expose an in-page debug log viewer; troubleshooting should use stored logs or backend diagnostics.

The image settings use a global concurrency limit, defaulting to 5, instead of hot/warm window controls; preload page count, global concurrency, and page offset keep sign/integer validation without artificial upper limits, while the visible matrix remains capped at 24 dots. The service worker starts reader preloading from EH/EX `/s/` pages and extracts reader links from `/g/` gallery pages. Reader-page preloading fetches the current reader page, reads real `#next` / `nexturl` and `#prev` / `prevurl` targets, then preloads in alternating order: next page, previous page, second next page, second previous page, continuing while `preloadAhead` leaves more slots. It follows each true `/s/` URL instead of synthesizing `token + pageNo`, and `globalConcurrency` controls how many reader page tasks can run at once. It resolves the main image URL, optionally stores the image body plus a data-URL delivery copy in the active cache store: default IndexedDB `ehp-preload-cache`, or the authorized directory when custom storage is enabled. Directory authorization acknowledges the picker message immediately; existing IndexedDB cache records are then copied to that directory in the background. The directory store writes identifiable reader images as `images/<gid>/<gid>-<pageNo>.<ext>`, keeps unclassified image files directly under `images/`, writes index JSON under `records/`, and mirrors state/settings/daily logs/statistics under `state/`, `settings/`, `logs/YYYY-MM-DD.json`, and `stats/`. Opening the directory store also migrates existing flat `images/` files and old `images/<gid>/page-<gid>-<pageNo>.<ext>` names into the gallery-folder layout and updates the affected main/resource record `directoryImageFile` values. If the image Blob fetch fails and the reader HTML exposes an EH/EX `nl()` token, the backend preload path uses the same replacement-page retry model as Super-preloader by fetching `pageUrl?nl=token`, parsing the replacement `#img` URL and next token, and retrying up to a small fixed limit; Super-preloader's reload button/UI behavior is not ported into the background cache path. Before each preload fetch is sent, the service worker queries the content script with `EHPE_PAGE_NETWORK_ACTIVITY_QUERY`; if the current page still has unfinished HTTP(S) image requests, only that not-yet-sent fetch waits and polls. Already-sent preload requests keep running, preload's own service-worker fetches do not pause other preload workers, and queued reader work is not removed just because the page is busy. The `优先取本地缓存` / `Prefer local cache` switch defaults off and controls the page-open/refresh cache-first consumer; only when explicitly enabled does it install the H@H network gate; when it is off, EH＋ does not hold, intercept, or replace reader/gallery images and only keeps observation/cache-fill behavior.

Native EH/EX reader navigation links such as `#prev`, `#next`, and page-jump links are no longer used as auto-pager detection signals. Active preload only gates each not-yet-sent fetch on current-page image request activity, while EH＋'s own built-in auto-pager continuing status remains a separate whole-run pause condition.

External image cache-fill is a default-on observation setting. When a gated preload fetch sees the page is idle, the content script maps current `#img`, `sp-exhentai-img-*`, and other loaded H@H/fullimg reader images with a resolvable `gid:pageNo` to permanent page keys before allowing that fetch to proceed. The service worker may then queue a low-priority HTTP-cache-first fetch of the already known image URL to write that image into its own cache. This path does not fetch `/s/` pages, does not depend on the `/s/` token for the storage key, and does not increment active preload queue/request counters. Images that only have a H@H resource URL may be stored as temporary resource-only cache and are excluded from the permanent index; temporary cache is removed after all E-Hentai / ExHentai tabs close, survives abnormal browser shutdown until the next normal all-tabs-closed cleanup, counts toward storage usage, and is evicted before permanent images when size limits are exceeded. Temporary cache hits still count as cache hits. Cleaning temporary cache removes only the image body and temporary index records; statistics remain valid historical counters and must not be decremented. This must not hook, intercept, or rewrite other plugins' fetch/XHR/Image/DOM paths.

The extension exposes an optional cooperative cache query interface for other extensions through `chrome.runtime.sendMessage(EHPE_EXTENSION_ID, { type: 'EHPE_COOPERATIVE_CACHE_QUERY', version: 1, ... })`. The content script also exposes the same query shape to page scripts on E-Hentai / ExHentai via `window.postMessage`; responses use `type: 'EHPE_COOPERATIVE_CACHE_RESPONSE'` and echo `requestId`. The query may identify exact local cache entries by `pageKey`, `gid` + `pageNo`, `/s/` page URL, `galleryKey`, `/g/` gallery URL, or image/resource URL. Resolved `/s/` pages are reader cache entries keyed by `gid:pageNo`; resolved `/g/` galleries are gallery cache entries keyed by `gid:token`; resource-only image URLs are temporary/resource cache candidates, not permanent page-index records. On hit, the response returns cache delivery metadata. Reader hits increment `/s/` reader reads and hits; gallery hits increment `/g/` reads and resource hits; resource-only image hits increment `/g/` resource hits, not `/s/` reader hits; misses do not change hit statistics. The current preloader writes reader records into the active cache store and exposes data-URL delivery through this protocol without cache enumeration. When `readerCacheFirstEnabled` is explicitly enabled, the extension's own `/s/` reader cache-first path uses a tab-scoped `declarativeNetRequest` gate, a `document_start` MAIN-world holder script, and the isolated content script: the gate redirects initial H@H image subresource loads to an extension placeholder while the isolated script queries the current `gid:pageNo` local cache before page-session preload startup work. During the local read, the placeholder shows `正在读取本地缓存...` with a looping one-dot to three-dot ellipsis animation. Internal reader cache-first queries use a memory-first fast response path and avoid immediate storage reads/state/log writes after cache records are loaded; external/page-bridge cooperative queries still use the normal counted/logged path. The isolated controller writes page-visible query diagnostics for `send-start`, `send-resolve`, `send-error`, `response-hit`, and `response-miss`, including elapsed time, response hit/reason/delivery kind, URL availability, runtime errors under `data-ehpe-cache-first-controller-*`, and network gate state under `data-ehpe-cache-first-network-block-*`. Hits keep the gate active for the reader page while the opt-in switch remains enabled and apply a local data/blob/extension URL; misses, errors, the 3500 ms fast-query timeout, or setting-disabled release the gate before restoring or allowing the original URL; later settings sync only updates the page-side enable hint and must not reinstall the gate for the same restored page. The 5-second timeout is only a visible soft `timeout-waiting` state; the 20-second isolated hard timeout, 30-second service-worker DNR auto-release, and 32-second MAIN-world restore timer are stalled-message watchdogs. `/g/` pages query existing `/s/` links and replace only their existing thumbnail `img` on exact local hits. By default `readerCacheFirstEnabled` is off. When it is off, these replacement checks do not run, and EH＋ only observes loaded images for cache-fill. These replacement checks do not fetch reader pages or images on miss.

Browsing history records `/g/` gallery and `/s/` reader visits as lightweight `recordKind: 'history'` entries in the active cache store, deduplicated by gid and capped by the history limit setting (default 100). The standalone `history.html` page opens from the floating panel statistics group or the popup, supports all / `/g/` / `/s/` source filtering, last-visited / visit-count sorting, per-entry delete, and clear-all. Visit-count sorting is the frequently watched view. History entries are excluded from cache cleanup, image limits, dedupe, cooperative cache hits, and `/s/` hit-rate statistics.

The account settings use only per-field toggles for the top account summary. Unchecking every account field hides the top summary, and old stored `accountStatusEnabled` values are ignored. Background account refresh uses `chrome.tabs` and `chrome.alarms`: while at least one E-Hentai / ExHentai `/g/` gallery or `/s/` reader tab is open, the service worker refreshes account status every 600 seconds; when the tracked open-tab count changes from zero to active, it refreshes once immediately. Closing or navigating away from all such tabs clears the repeating refresh alarm.

Image settings include both frequent image protection and frequent `/g/` gallery protection. Each protection type has its own enable switch and read-count threshold. Cleanup skips protected images/galleries by default for age-based cleanup and provides separate include-protected checkboxes for images and galleries. In age-based cleanup, `cleanupDays=0` disables time-based cache cleanup instead of matching all cache.

The statistics tab no longer has a list filter or embedded frequently watched list. The `/s/` and `/g/` statistics remain visible as separate per-field rows, and each row checkbox independently controls whether that statistic appears in the top mini summary.

The colors group lists every dot state and its current color. Users can click the dot icon, use the color picker, or enter hex values such as `ff`, `0ff`, `00e5ff`, or `#00e5ff`; the row previews the pending color first, then applies and saves it only after Confirm. Cancel or clicking elsewhere restores the saved color.

The floating panel also includes an about group. It shows the current extension version, a GitHub icon/link, GitHub-only source notice, free/open-source notice, and a warning that paid copies elsewhere are third-party resale of open source. The project GitHub action opens the real repository, and update checks query the project Release API. Automatic update checks run at most once every 7 days, and available updates show a small clickable notice beside the floating-panel title. The current 1.0.1 build does not simulate an available update for UI inspection.
